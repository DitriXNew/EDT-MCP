# create_extension_project

Create a NEW 1C configuration extension project in the EDT workspace bound to a base configuration project. The extension project is immediately available for metadata adoption (adopt_metadata_object) and BSL override. The name must not already exist as a project. standardChecks/commonChecks are applied only when com.e1c.v8codestyle is installed. Full parameters and examples: call get_tool_guide('create_extension_project').

## Parameters
| Parameter | Required | Type | Description |
| --- | --- | --- | --- |
| name | yes | string | Name of the new extension Configuration object. Also used as the default EDT project name if projectName is not supplied. |
| baseProjectName | yes | string | Name of the BASE configuration EDT project the extension will extend. Must be an existing, open V8 configuration project. Use list_projects to find it. |
| projectName | — | string | EDT workspace project name to create. Defaults to `<baseProjectName>.<name>` when omitted. |
| prefix | — | string | NamePrefix for the extension (default empty string). |
| synonym | — | string | Human-readable synonym for the extension Configuration. Defaults to the `name` value if omitted. |
| comment | — | string | Optional free-text comment set on the extension Configuration. |
| purpose | — | string | Extension purpose: Customization (default), AddOn, or Patch. |
| compatibilityMode | — | string | Optional CompatibilityMode enum literal (e.g. `Version8_3_10`); empty = factory default. |
| standardChecks | — | boolean | Enable 1C:Standards BSL checks (default true). Applied only when com.e1c.v8codestyle is installed. |
| commonChecks | — | boolean | Enable common project-level BSL checks (default true). Applied only when com.e1c.v8codestyle is installed. |
| autoSortTopObjects | — | boolean | Reserved for future use; accepted but not yet applied in this release. |

## Guide
Wraps the EDT "New Extension" project-creation wizard. Creates a brand-new extension
project in the workspace, bound to a base configuration, ready for metadata adoption.

## When to use

- You need a configuration extension project to override metadata objects (use
  `adopt_metadata_object` after creation) or add BSL interceptors.
- The `name` must not already exist as a workspace project (the tool rejects duplicates).
- The base project must be an open, V8 configuration project (not an extension).

## Workflow

1. `create_extension_project` — create the extension and bind it to the base.
2. `adopt_metadata_object` — adopt the objects/members you want to override into the extension.
3. `create_metadata` / `modify_metadata` on the adopted objects — extend the behavior.
4. `delete_project` — remove the extension project when it is no longer needed.

## Gotchas

- **Extension name must be new.** The call is rejected if a workspace project with the
  computed name already exists. Use a unique `name` or pass an explicit `projectName`.
- **Base project must be a configuration, not an extension.** Pass the BASE
  configuration's project name.
- **After creation, wait for `state=ready`** before calling `adopt_metadata_object`.
- **autoSortTopObjects is not yet applied.** See `codestyle.autoSortNote` in the response.
