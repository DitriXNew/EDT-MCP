# Совместимость с версиями EDT (дрейф API)

API EDT меняется между релизами. Эта глава фиксирует, как с этим жить.

## Доказанный пример дрейфа

Сигнатура `IFormGenerator.generateForm(...)` (пакет `com._1c.g5.v8.dt.form.generator`):

| Релиз / плагин | Версия пакета | Сигнатура `generateForm` |
|---|---|---|
| `com._1c.g5.v8.dt.form_29.0.0` (≈EDT 2025.1, июнь) | `3.1.0` | 8 параметров, **без** `InterfaceCompatibilityMode` |
| `com._1c.g5.v8.dt.form_31.1.4` (EDT 2025.1.5) | `4.0.0` | 9 параметров, **добавлен** `InterfaceCompatibilityMode` |

Вывод: между релизами происходит **ломающее** изменение — мажорная версия пакета
`3.1.0 -> 4.0.0` и добавление параметра в публичный метод. Код, скомпилированный под одну
сигнатуру, не слинкуется/не вызовется на другой версии без специальных мер.

## Где именно происходят изменения (уровни)

1. **Версия экспортируемого пакета** (`...;version="4.0.0"`). Контролирует, против какой
   EDT бандл вообще резолвится (OSGi `Import-Package` с диапазоном версий).
2. **Сигнатура метода / набор классов**. Резолв прошёл, но прямой вызов падает
   `NoSuchMethodError` / `LinkageError`, если число/типы параметров изменились.
3. **Поведение / значения enum**. Реже, но возможны новые/убранные значения (`FormType`
   стабилен — 19 значений в обеих версиях, но это не гарантия для других enum).

## Приёмы, уже применяемые в этом бандле

### 1. Диапазоны версий в `Import-Package`

В `META-INF/MANIFEST.MF` зависимости импортируются с диапазоном, например:

```
com._1c.g5.v8.dt.core.platform;version="[12.0.0,13.0.0)"
```

Это жёстко ограничивает совместимые версии EDT и даёт **внятный отказ резолва** вместо
падения в рантайме. Минус: при выходе мажорной версии пакета бандл перестаёт
устанавливаться, пока диапазон не расширят.

### 2. Доступ к API через рефлексию по строковому имени класса

Так сделано для CLI- и LanguageTool-API в `Activator` (трекеры по `String` class name) —
бандл **не зависит на этапе сборки** от изменчивых/опциональных плагинов, а вызовы идут
через рефлексию. Применимо, когда API может отсутствовать или менять пакет.

### 3. Толерантный поиск метода

`com.ditrix.edt.mcp.server.utils.ReflectionUtils.findMethod(Class, name, paramTypes...)` —
ищет метод, переживая часть изменений. На его основе можно делать «перебор перегрузок».

### 4. Определение версии EDT в рантайме

`GetEdtVersionTool.getEdtVersion()` (через `eclipse.buildId` / product bundle / RCP-бандл)
и `com._1c.g5.v8.dt.platform.version.Version` (из `IV8Project.getVersion()` — версия
платформы 1С, уже используется в `CreateMetadataObjectTool`). Позволяет ветвить логику.

## Рекомендуемая стратегия для волатильного API (например `generateForm`)

Комбинация из трёх слоёв:

1. **Изолировать вызов за фасадом.** Один адаптер-класс (например `FormGeneratorAdapter`)
   инкапсулирует обращение к `IFormGenerator`. Весь остальной код инструмента не знает о
   версионных различиях.
2. **Рефлективный вызов с перебором сигнатур.** Внутри фасада не звать `generateForm`
   напрямую, а найти подходящую перегрузку: сначала 9-параметровую (новые EDT), при
   отсутствии — 8-параметровую (старые). Пример:

```java
Method m = ReflectionUtils.findMethod(gen.getClass(), "generateForm",
        MdObject.class, BasicForm.class, FormType.class, ScriptVariant.class,
        String.class, Version.class, FormFieldInfo.class, Integer.class,
        InterfaceCompatibilityMode.class);
if (m != null) {
    return m.invoke(gen, owner, basicForm, type, sv, name, version, fields, columns, mode);
}
m = ReflectionUtils.findMethod(gen.getClass(), "generateForm",
        MdObject.class, BasicForm.class, FormType.class, ScriptVariant.class,
        String.class, Version.class, FormFieldInfo.class, Integer.class);
if (m != null) {
    return m.invoke(gen, owner, basicForm, type, sv, name, version, fields, columns);
}
throw new IllegalStateException(
    "Unsupported EDT form generator API; tested 2025.1.x. EDT version: " + GetEdtVersionTool.getEdtVersion());
```

3. **Внятная деградация.** Если ни одна сигнатура не найдена — вернуть ошибку инструмента
   с указанием версии EDT (как сделано для отсутствующего LanguageTool), а не уронить
   плагин `LinkageError`-ом.

> Компромисс: типы-аргументы (`FormType`, `InterfaceCompatibilityMode`, `ScriptVariant`)
> всё равно нужны на этапе сборки. Если меняется и пакет с этими типами, фасад целиком
> переводят на рефлексию (типы — по `Class.forName`/enum `valueOf`), ценой читаемости.

## Организационные меры

- **Базовая версия EDT.** Зафиксировать в `README.md` папки минимально/максимально
  проверенную версию (сейчас — `2025.1.5`). Каждую запись API помечать версией источника.
- **Матрица проверки.** При апгрейде EDT прогонять smoke-тест инструментов на новой
  версии (генерация форм элемент/список/выбор + `get_project_errors`).
- **Локальный «барьер версий».** В начале volatile-инструмента проверять
  `GetEdtVersionTool.getEdtVersion()` и предупреждать, если версия вне протестированного
  диапазона (не блокировать, а помечать как непроверенную).
- **Хранить декомпил-снимки сигнатур.** Эта папка `docs_api-edt` и есть такой реестр:
  при смене релиза повторно снять `javap` ключевых классов и сравнить (как в таблице выше).

## Быстрый чек-лист при смене релиза EDT

1. `javap -p` ключевых интерфейсов (`IFormGenerator`, `IFormItemManagementService`, ...)
   из нового `.jar`, сравнить с зафиксированным здесь.
2. Сверить версии экспортируемых пакетов; при бампах — расширить диапазоны в `Import-Package`.
3. Проверить, что рефлективные фасады находят новую перегрузку (или добавить ветку).
4. Прогнать smoke-тест инструментов.
5. Обновить версию в `README.md` и таблицы сигнатур в соответствующих главах.
